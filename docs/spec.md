# Техническое задание: StealthChat

## 1. Общие положения

**Основная цель:** Создать полностью децентрализованный, анонимный мессенджер для личного общения (1-на-1) с возможностью групповых чатов, работающий в условиях активного DPI и блокировок. Трафик приложения должен быть неотличим от обычного веб-трафика (HTTPS/WebSocket, порт 443).

**Ключевые принципы:**
- Децентрализация: нет единого сервера, любой узел может выступать ретранслятором
- Анонимная регистрация: свободный логин + пароль, без email и телефонов
- E2E-шифрование всех сообщений и файлов (Double Ratchet)
- Ретрансляторы не хранят сообщения и метаданные
- Файлы хранятся исключительно на конечных устройствах
- Сетевое взаимодействие полностью скрыто под видом HTTPS/WSS-трафика, порт 443
- Всплывающие уведомления на всех платформах

## 2. Анти-DPI требования (критически важно)

1. **Транспорт:** только TCP, порт 443. Никакого UDP-трафика, никаких нестандартных портов.
2. **Протокол:** WebSocket Secure (WSS) поверх TLS 1.3 с валидными сертификатами.
3. **TLS-отпечаток:** клиент должен имитировать браузерный TLS-стек. Использовать системные TLS-библиотеки без модификаций.
4. **Размер пакетов:** все сообщения перед шифрованием дополняются до ближайшего фиксированного размера (512 байт) с добавлением случайного паддинга.
5. **Временное распределение:** в отсутствие реального трафика отправляются dummy-пакеты (интервалы от 10 до 60 секунд).
6. **Обнаружение пиров:** полностью исключается выделенный DHT-трафик (UDP). Вся маршрутная информация передаётся внутри зашифрованных WSS-каналов (gossip-протокол).
7. **Сокрытие метаданных:** если включён onion-режим, ретранслятор не знает конечного адресата и отправителя.
8. **Отсутствие открытого текста в рукопожатиях:** вся информация передаётся внутри зашифрованного туннеля после установки WSS-соединения.

## 3. Функциональные требования

### 3.1. Регистрация и идентификация
- Логин (псевдоним) + пароль. Пароль используется для расшифровки локального профиля (Argon2id → ключ).
- Генерируется долговременная пара ключей Ed25519.
- Публичный ключ – уникальный идентификатор (peer ID).
- Контакты добавляются путём обмена публичными ключами (QR-код или текстовая строка).

### 3.2. Личные чаты (1-на-1)
- Каждая беседа – отдельная сессия Double Ratchet с прямой секретностью.
- Поддерживаются: текст, изображения, файлы (до 500 МБ).

### 3.3. Сетевая архитектура (анти-DPI)
- Все узлы являются одновременно клиентами и ретрансляторами (можно отключить роль ретранслятора).
- Узел подключается к одному или нескольким соседним узлам через WSS на порту 443.
- Соединения постоянно поддерживаются (persistent WebSocket).

**Peer Discovery (поиск собеседника):**
- Список известных узлов (seed-пиров) хранится локально.
- Когда Алиса хочет связаться с Бобом, она запрашивает у уже подключённых соседей информацию о том, знают ли они адрес Боба (по его публичному ключу).
- Запрос перенаправляется дальше с TTL, подобно поиску в распределённой хеш-таблице, но без отдельного протокола – всё внутри WSS.

**Ретрансляция:** все сообщения передаются от отправителя к получателю через цепочку WSS-туннелей.

### 3.4. Onion-маршрутизация (опционально)
- Если включена, между Алисой и Бобом строится цепочка из трёх узлов.
- Трафик внутри каждого WSS-туннеля неотличим от обычного HTTPS.
- Длина цепочки фиксирована (3).

### 3.5. Интеграция с Yggdrasil (альтернативный транспорт)
- Если в системе запущен Yggdrasil, приложение может использовать его IPv6-адреса для WSS-соединений.

### 3.6. Уведомления
- Android: foreground service с обязательным WSS-подключением.
- Linux/Windows: фоновый процесс, уведомления через системные API.

## 4. Нефункциональные требования

- **Скрытность:** весь трафик должен классифицироваться DPI как «Generic HTTPS» или «WebSocket over TLS».
- **Надёжность:** автоматическое переключение ретрансляторов без потери сессий.
- **Задержка:** доставка текстового сообщения не более 2 секунд при живом соединении.
- **Масштабируемость:** архитектура рассчитана на сотни тысяч узлов.
- **Автономность:** после первоначальной настройки seed-пиров приложение работает без внешней инфраструктуры.

## 5. Технологический стек

- **Язык:** Dart + Flutter (Android, Linux, Windows).
- **Криптография:** `libsodium` (через FFI) – Ed25519, X25519, AEAD ChaCha20-Poly1305.
- **Сеть:**
  - WebSocket Secure (WSS) через стандартную платформенную реализацию TLS.
  - Библиотека для WebSocket: `web_socket_channel`.
- **Хранилище:** SQLite через `sqflite_sqlcipher` (шифрованная БД).
- **Обфускация трафика:**
  - Padding: перед шифрованием каждого сообщения добавляется `pad_len` (0–510 байт), чтобы общая длина была кратна 512.
  - Dummy-трафик: отдельный поток, генерирующий случайные WSS-кадры.
  - TLS fingerprint: использование системного HTTP-клиента с настройками по умолчанию.

## 6. Архитектурные компоненты

- **IdentityManager:** создание и хранение ключей, локальная аутентификация.
- **CryptoSession:** управление Double Ratchet сессиями.
- **PeerDiscovery:** gossip-логика обмена таблицами маршрутизации внутри WSS-соединений.
- **RelayPool:** поддержание пула активных WSS-соединений к соседним узлам, выбор маршрутов.
- **MessageRouter:** сериализация, шифрование, паддинг, onion-инкапсуляция и отправка через RelayPool.
- **TrafficObfuscator:** генерация dummy-кадров, поддержание паттернов трафика.
- **FileTransfer:** чанковая передача с контролем целостности.
- **NotificationService:** платформенные уведомления.

## 7. Порядок реализации

### Этап 1 – Локальная аутентификация
1. Базовое Flutter-приложение, экраны входа.
2. IdentityManager: создание Ed25519-ключей, шифрование хранилища.

### Этап 2 – WSS-туннели и gossip
3. WebSocket-сервер (внутри узла) с поддержкой TLS (самоподписанный сертификат).
4. WebSocket-клиент, подключение к seed-узлам.
5. Протокол обмена маршрутной информацией (peer discovery) внутри WSS.

### Этап 3 – E2E-шифрование 1-на-1
6. Реализация Double Ratchet, первоначальный обмен ключами.
7. Отправка текстовых сообщений с шифрованием.

### Этап 4 – Анти-DPI обфускация
8. Реализация паддинга сообщений (блок 512 байт).
9. Генератор dummy-трафика.
10. Проверка TLS-отпечатка: использовать системный стек без изменений.

### Этап 5 – Onion-маршрутизация
11. Построение onion-цепочек, слойное шифрование.
12. Интеграция в MessageRouter.

### Этап 6 – Файлы и уведомления
13. Потоковая передача файлов, чанкинг.
14. Платформенные всплывающие уведомления.
15. Поддержка офлайн-буферизации.

### Этап 7 – Дополнительно
16. Интеграция с Yggdrasil.
17. Пользовательский интерфейс: чаты, контакты, настройки.
18. Групповые чаты.

## 8. Протокол маскировки трафика (спецификация)

- **Фаза 1. Установка WSS соединения.**
  - Клиент инициирует стандартное TLS-рукопожатие (порт 443).
  - Сервер предъявляет самоподписанный сертификат с публичным ключом.
  - Клиент проверяет, что ключ соответствует ожидаемому для данного peer ID.

- **Фаза 2. WebSocket Upgrade.**
  - Внутри TLS выполняется стандартный HTTP Upgrade до WebSocket.
  - Заголовки имитируют обычный браузер.

- **Фаза 3. Инкапсуляция сообщений.**
  - Каждый кадр WebSocket имеет тип Binary.
  - Полезная нагрузка = `[1 байт типа сообщения][зашифрованный блок]`.
  - Типы: 0x01 – реальные данные, 0x02 – dummy, 0x03 – gossip-запрос.
  - Все блоки дополняются до размера, кратного 512 байт.

- **Фаза 4. Поддержание сессии.**
  - Если настоящих сообщений нет, каждые 10–30 секунд отправляется dummy-кадр (тип 0x02) размером 512 байт.

## 9. Требования к коду

- Минимальные внешние зависимости.
- Обязательное тестирование обфускации с анализатором трафика.
- Документированный процесс настройки seed-узлов.
